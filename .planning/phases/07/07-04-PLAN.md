  <files>apps/web/src/services/integrations/gmail.ts, apps/web/src/services/integrations/outlook.ts</files>
  <action>
Extend both provider send functions to support CC and BCC fields:

1. In `gmail.ts`, update `SendEmailInput` interface:
   ```typescript
   export interface SendEmailInput {
     to: string[];
     cc?: string[];
     bcc?: string[];
     subject: string;
     body: string;
     replyToThreadId?: string;
     inReplyTo?: string; // Message-ID header for proper threading
     references?: string; // References header for thread chain
   }
   ```
   Update the MIME message builder to include CC and BCC headers:
   ```
   Cc: ${input.cc?.join(", ")}
   Bcc: ${input.bcc?.join(", ")}
   ```
   Only add these headers if the arrays are non-empty. Gmail handles BCC by stripping the header after sending.

   Also add `In-Reply-To` and `References` headers when `replyToThreadId` is provided. This ensures reply emails thread correctly in the recipient's inbox (not just in Gmail's thread view).

2. In `outlook.ts`, update `SendEmailInput` similarly. Add CC/BCC to the Graph API payload:
   ```typescript
   ccRecipients: input.cc?.map(email => ({ emailAddress: { address: email } })),
   bccRecipients: input.bcc?.map(email => ({ emailAddress: { address: email } })),
   ```

3. Update both send functions to also record `ccEmails` in the `email_messages` insert (the schema already has a `ccEmails` column).

4. In the `email_messages` insert for both providers, link the sent message to a `recordId` if one is passed through. Currently, sendEmail doesn't accept a recordId — add it as an optional parameter or handle it in the caller (plan 04 task 2 will pass it via the service layer).

Per PITFALLS.md Pitfall 5: Both send functions ALREADY use the user's OAuth token to send through the provider's own API. Do NOT change this to use Resend or any SMTP relay. Resend is only for system notifications (invites, etc.).

2. Create `apps/web/src/app/api/v1/records/[recordId]/send-email/route.ts`:
   - POST handler:
     ```typescript
     export async function POST(req: NextRequest, { params }: { params: Promise<{ recordId: string }> }) {
       const ctx = await getAuthContext(req);
       if (!ctx) return unauthorized();
       const { recordId } = await params;
       const body = await req.json();
       // Validate: to required (array, non-empty), subject required, body required
       // cc and bcc optional arrays
       const result = await sendEmailFromRecord(ctx.workspaceId, ctx.userId, recordId, body);
       return success(result);
     }
     ```
   - Return 400 for validation errors and missing provider connection
   - Return 502 for provider API errors

Note: The existing `/api/v1/integrations/gmail/send` and `/api/v1/integrations/outlook/send` endpoints remain for direct provider sends. The new `/api/v1/records/:recordId/send-email` is the preferred endpoint for compose-from-record because it auto-detects provider and links to the record.
