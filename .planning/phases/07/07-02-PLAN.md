  <files>apps/web/src/services/integrations/google-calendar.ts, apps/web/src/services/integrations/outlook-calendar.ts, apps/web/src/app/api/v1/cron/calendar-sync/route.ts</files>
  <action>
The calendar sync and meeting_ended signal emission already work. This task adds defensive hardening:

1. In `google-calendar.ts`, modify `syncCalendarEvents()`:
   - Wrap the `calendar.events.list()` call. If the API returns a 410 (syncToken expired), clear the syncToken from providerMetadata and restart with a 30-day lookback (same as initial sync path). Log `[google-calendar] syncToken expired — performing 30-day re-sync`.
   - Add try/catch around individual event processing so one bad event doesn't stop the entire sync batch.

2. In `outlook-calendar.ts`, modify `syncCalendarEvents()`:
   - Handle HTTP 410/syncStateNotFound by clearing the syncCursor and restarting with 30-day initial delta. Log `[outlook-calendar] deltaToken expired — performing 30-day re-sync`.
   - Add per-event try/catch for resilience.

3. In `cron/calendar-sync/route.ts`:
   - Add Outlook Calendar subscription existence check: if `providerMetadata.calendarSubscriptionId` is missing, call `subscribeToCalendarNotifications()` to create one (not just check expiry).
   - Apply the same 18-hour buffer for subscription renewal (change from 12-hour to 18-hour check).
   - Add Google Calendar watch renewal: check if Google push notification channel needs re-registration. Google Calendar push channels expire after 7 days. If `providerMetadata.calendarWatchExpiry` is within 24 hours or missing, call `watchCalendar()`.
   - Update `watchCalendar()` in google-calendar.ts to store `calendarWatchExpiry` in providerMetadata after successful registration.
   - Add error tracking: update `errorMessage` on failure, clear on success. Skip tokens with `status: error`.

4. In `processCalendarEventLifecycle()` for both providers:
   - Add a guard: only emit `meeting_ended` for events with a linked `recordId`. The code already does this (`if (event.recordId)`), but also log a counter of unmatched events for observability: `[calendar] Processed N ended events, M matched to records`.

   ```typescript
   /**
    * Shared email-to-record matching for all integrations.
    * Finds CRM records (People, Companies, Deals) that have an email attribute
    * matching any of the provided email addresses.
    */
   export async function matchEmailsToRecords(
     workspaceId: string,
     emails: string[]
   ): Promise<string | null>
   ```
   - Move the shared logic from gmail.ts `matchEmailToPeopleRecord()` into this file
   - Add workspace scoping: join through `records` -> `objects` to verify the matched record belongs to the correct workspace (the existing code has a comment noting this gap)

   - Return the first match (keep existing behavior)

2. Update all four files (gmail.ts, outlook.ts, google-calendar.ts, outlook-calendar.ts):
   - Remove the local `matchEmailToPeopleRecord()` / `matchEmailsToRecord()` functions
   - Import from `./email-matching`
   - Update call sites to use the new function name `matchEmailsToRecords()`

This deduplication is important because email matching is a critical integration point (SYNC-04) and having four divergent copies means bugs in matching only get fixed in one place.
