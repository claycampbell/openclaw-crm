<objective>
Harden Gmail and Outlook delta sync with error recovery and proactive maintenance.

Purpose: The sync services already work for the happy path but lack resilience. Gmail sync breaks irrecoverably when historyId expires (>30 days or mailbox changes). Outlook sync silently stops when delta tokens expire after 7 days of inactivity. Both need defensive recovery paths before the email compose UI (plans 03-05) can rely on sync data being present.

Output: Hardened Gmail sync with bounded partial-sync recovery, proactive Outlook delta token refresh, and webhook subscription lifecycle management.
</objective>

In `gmail.ts`, modify `syncNewMessages()` to handle historyId invalidation:

1. Wrap the `gmail.users.history.list()` call in a try/catch. When the API returns a 404 with `historyId is not valid` (or a 410 Gone), instead of throwing, call a new `boundedPartialSync()` function.

2. Create `boundedPartialSync(workspaceId, userId, gmail)`:
   - Uses `gmail.users.messages.list({ userId: "me", q: "after:YYYY/MM/DD" })` where the date is 7 days ago
   - Fetches metadata for each message (same as the existing delta sync path)
   - Deduplicates via `markSignalProcessed()` (same as existing)
   - Inserts into `email_messages` and emits signals (same as existing)
   - After processing, calls `gmail.users.getProfile({ userId: "me" })` to get the current historyId from the profile response (historyId field)
   - Stores the fresh historyId as the new `syncCursor`
   - Logs `[gmail] historyId invalid — recovered via 7-day partial sync, processed N messages`

3. Add per-user error tracking: if `syncNewMessages` fails 3 times consecutively for the same user (track via `errorMessage` field on `integrationTokens`), update the token status to `error` and set `errorMessage` to the failure reason. Do NOT attempt sync for tokens with `status: error` — skip them in the cron.

4. In `cron/gmail-sync/route.ts`, add a filter to skip tokens with `status: error` (already filtered by `eq(integrationTokens.status, "active")`  so this is already handled, but add a log line noting skipped error tokens for observability). Also add a `try/catch` around each token's sync that updates `errorMessage` on failure and clears it on success.

Key pitfall to avoid (from PITFALLS.md Pitfall 2): Do NOT do a full inbox scan on historyId failure. The bounded 7-day window prevents quota exhaustion. Apply a rate limit of max 50 messages per partial sync to avoid hitting Gmail API quotas.

   - Calls `syncNewMessages()` but with a purpose of keeping the delta token fresh

   - Store the new deltaLink as syncCursor
   - Log `[outlook] Delta token expired — recovered via 7-day catch-up, N messages`

2. Modify `syncNewMessages()` to handle `syncStateNotFound` error (HTTP 410 from Graph). When this error occurs, call the bounded recovery path (same as above). Extract this into a shared helper `recoverDeltaSync()`.

3. Add proactive subscription renewal: In the cron endpoint, for each Outlook token, check `providerMetadata.subscriptionExpiry`. 5. Apply the same error tracking pattern as Gmail Task 1: track consecutive failures in `errorMessage`, skip tokens in error state.
