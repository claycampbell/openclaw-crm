- Stores the rows in the job payload (JSONB). For very large payloads (>5MB), this is acceptable since PostgreSQL JSONB handles up to 1GB. The alternative (temp file storage) adds complexity without benefit at CRM scale.
- Enqueues an `import_records` job via enqueueJob
- Returns the job ID for progress polling

- Queries background_jobs by ID
- Returns { status, payload.progress } where progress is { processed: number, total: number, created: number, updated: number, skipped: number, errors: number }

**Update the import_records job handler** -- register in `apps/web/src/instrumentation.ts`:
```typescript
registerJobHandler("import_records", async (payload) => {
  const { workspaceId, objectId, rows, dedupStrategy, dedupAttributeSlug, userId } = payload as ImportJobPayload;
  // Process in batches of 100 for progress updates
  const batchSize = 100;
  const total = rows.length;
  let processed = 0;
  let created = 0, updated = 0, skipped = 0;
  const errors: { row: number; message: string }[] = [];

  // Build dedup lookup once before processing
  const lookup = (dedupStrategy !== "create_all" && dedupAttributeSlug)
    ? await buildDuplicateLookup(workspaceId, objectId, [dedupAttributeSlug])
    : null;

  for (let i = 0; i < total; i += batchSize) {
    const batch = rows.slice(i, i + batchSize);
    const result = await importRecordsWithDedup({
      workspaceId, objectId, rows: batch,
      dedupStrategy, dedupAttributeSlug, userId,
      existingLookup: lookup, // Pass pre-built lookup to avoid rebuilding per batch
    });
    created += result.created;
    updated += result.updated;
    skipped += result.skipped;
    errors.push(...result.errors.map(e => ({ ...e, row: e.row + i })));
    processed += batch.length;

    // Update job payload with progress (update the background_jobs row)
    await updateJobProgress(jobId, { processed, total, created, updated, skipped, errors: errors.length });
  }
});
```

Add `updateJobProgress(jobId, progress)` helper that updates the payload JSONB with a `progress` key.
