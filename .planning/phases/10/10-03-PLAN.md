Add the following functions to `apps/web/src/services/webhooks.ts` (extending the CRUD from Plan 01):

1. **dispatchWebhookEvent(workspaceId: string, eventType: string, payload: object)**
   - Query webhook_subscriptions WHERE workspace_id = workspaceId AND enabled = true AND events array contains eventType
   - For each matching subscription:
     a. Insert a row into webhook_deliveries (subscription_id, event_type, payload, status: "pending")
     b. Enqueue a `webhook_send` job with { deliveryId, subscriptionId } via enqueueJob
   - Fire-and-forget (wrap in try/catch, never throw -- webhook failures must not block CRM operations)

2. **sendWebhookDelivery(deliveryId: string)**
   - Load the delivery row + its subscription (join)
   - If subscription is disabled, mark delivery as "dead_letter" and return
   - Build the request body: `{ event: eventType, data: payload, timestamp: ISO string, delivery_id: deliveryId }`
   - If subscription has a secret, compute HMAC-SHA256 signature:
     ```typescript
     const signature = crypto.createHmac("sha256", subscription.secret).update(JSON.stringify(body)).digest("hex");
     ```
   - POST to subscription.url with headers:
     - `Content-Type: application/json`
     - `X-Webhook-Signature: sha256={signature}` (omit if no secret)
     - `X-Webhook-Event: {eventType}`
     - `X-Webhook-Delivery: {deliveryId}`
   - Set a 10-second timeout on the fetch call (use AbortController)
   - On success (2xx response):
     - Update delivery: status = "success", response_code, response_body (first 1KB), completed_at, attempts + 1
     - Reset subscription.consecutive_failures to 0
   - On failure (non-2xx, timeout, network error):
     - Update delivery: attempts + 1, response_code (if available), error_message
     - Increment subscription.consecutive_failures
     - **Circuit breaker check**: if consecutive_failures >= 5:
       - Set subscription.enabled = false, disabled_reason = "Circuit breaker: 5 consecutive delivery failures"
       - Mark all pending deliveries for this subscription as "dead_letter"
       - Create a notification for the subscription creator: "Webhook to {url} has been disabled after 5 consecutive failures"
     - If attempts < max_attempts AND circuit not tripped:
       - Compute next retry: exponential backoff with jitter
         - Attempt 2: 1 min + random 0-30s
         - Attempt 3: 5 min + random 0-60s
         - Attempt 4: 30 min + random 0-120s
       - Update delivery: next_retry_at, status stays "pending"
     - If attempts >= max_attempts:
       - Update delivery: status = "failed", completed_at

3. **processWebhookRetries(batchSize: number = 20)**
   - Query webhook_deliveries WHERE status = "pending" AND next_retry_at <= NOW() AND next_retry_at IS NOT NULL, limit batchSize
   - For each, enqueue a `webhook_send` job

Register job handler in `apps/web/src/instrumentation.ts`:
```typescript
registerJobHandler("webhook_send", async (payload) => {
  const { deliveryId } = payload as { deliveryId: string };
  await sendWebhookDelivery(deliveryId);
});
```

Also add a periodic retry check -- either add `processWebhookRetries()` to the existing cron endpoint at `/api/v1/cron/jobs`, or create a dedicated cron route at `/api/v1/cron/webhook-retries`. The cron route pattern is preferred (separate concerns). The cron calls processWebhookRetries() which re-enqueues pending retries as jobs.
