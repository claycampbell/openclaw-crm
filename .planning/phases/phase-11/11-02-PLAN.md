Upgrade `getNextBestAction()` to incorporate signal_events and win/loss patterns. Keep the existing `NextBestAction` interface shape (no breaking changes). Add a new field `signalContext: string | null` to the interface (additive).

Changes to the service:

1. **Signal activity assessment**: After validating the record, query `signal_events` for this record_id in the last 30 days. Compute:
   - `totalSignals`: count of all signals
   - `lastSignalDate`: most recent signal's createdAt
   - `daysSinceLastSignal`: days since lastSignalDate (replaces the current notes/tasks-only calculation)
   - `signalTypes`: Set of unique signal types (email_received, meeting_ended, etc.)
   - `emailCount`: count of email_received + email_sent signals
   - `meetingCount`: count of meeting_ended signals

2. **Win/loss pattern context**: Import `hasMinimumDataVolume` from win-loss.ts. If sufficient data exists, import `getWinLossPatterns` and extract the top 2 patterns. Pass these patterns as context to the AI prompt so NBA suggestions are grounded in what actually predicts wins. For example: if "High engagement (4+ notes)" has a higher win rate, the AI can suggest adding more notes. Use the analytics cache to avoid re-computing win/loss on every NBA call.

3. **Enhanced urgency calculation**: Update urgency logic:
   - `high`: daysSinceLastSignal > 7 OR (stage is Negotiation/Contract Sent AND daysSinceLastSignal > 3)
   - `medium`: daysSinceLastSignal 3-7 days OR stage is early (Discovery/Qualified) with no meetings
   - `low`: active engagement (signal in last 3 days)

4. **Richer AI prompt**: Update `generateNBAFromAI` to include:
   - Signal activity summary: "12 emails, 3 meetings, last activity 2 days ago"
   - Win/loss context: "Top win patterns: deals with 4+ notes close at 72% vs 45% overall"
   - Existing note titles and task list (keep these)
   - Stage playbook candidates (keep these)
   - Ask AI to return the same JSON shape but ground its suggestion in the signal/pattern data

5. **Fallback behavior**: If no signal_events exist for this deal, fall back to the current behavior (notes + tasks + playbook). The service must work identically for workspaces with no signal data.

6. **Keep the in-memory 5-minute cache** for individual deal NBAs (already exists). This is separate from the analytics cache used for aggregate reports.

Do NOT change the function signature `getNextBestAction(workspaceId, recordId)`. The API route and component call this directly.
