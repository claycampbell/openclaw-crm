Rewrite the `processJobs()` function in `apps/web/src/services/job-queue.ts` to actually execute registered handlers:

1. Replace the naive SELECT + UPDATE with a single transaction using raw SQL `FOR UPDATE SKIP LOCKED`:
   ```typescript
   import { sql } from "drizzle-orm";

   // Inside processJobs(), use db.execute() with raw SQL:
   const claimed = await db.execute(sql`
     UPDATE background_jobs
     SET status = 'running', started_at = NOW()
     WHERE id IN (
       SELECT id FROM background_jobs
       WHERE status = 'pending' AND run_at <= NOW()
       ORDER BY run_at
       LIMIT ${batchSize}
       FOR UPDATE SKIP LOCKED
     )
     RETURNING *
   `);
   ```

2. For each claimed job, call `executeJob(job.type, { ...job.payload, workspaceId: job.workspace_id })`. If `executeJob` returns false (no handler registered), log a warning but still mark completed.

3. On success, mark status = 'completed' with completedAt = new Date().

4. On failure (catch block), implement exponential backoff:
   - Calculate `newRetries = Number(job.retries) + 1` (handle NaN: default to 1)
   - If `newRetries >= 3`: set status = 'failed', errorMessage = error string, completedAt = new Date() (dead-letter)
   - If `newRetries < 3`: set status = 'pending', calculate next runAt with exponential backoff: `new Date(Date.now() + Math.pow(2, newRetries) * 60_000)` (2min, 4min), set retries = String(newRetries), errorMessage = error string

5. Keep the existing `enqueueJob`, `registerJobHandler`, and `executeJob` functions unchanged (they work correctly).

6. In `apps/web/src/db/schema/jobs.ts`: No schema changes needed. The `retries` column is text which is unfortunate but functional -- handle the text-to-number conversion carefully in processJobs (parseInt with fallback to 0).
